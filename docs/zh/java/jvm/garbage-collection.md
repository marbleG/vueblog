# JVM 垃圾回收

## 垃圾回收基本概念

### 哪些内存需要回收？

JVM 垃圾回收主要回收**Java堆**和**方法区**中不再被引用的对象。

程序计数器、虚拟机栈、本地方法栈是线程私有的，随线程生随线程灭，不需要回收。

### 如何判断对象是否死亡？

**引用计数法**：
- 给对象添加一个引用计数器，被引用计数器+1，引用失效计数器-1
- 缺点：无法解决循环引用问题

**可达性分析算法**（HotSpot 使用）：
- 从 GC Roots 作为起点向下搜索，搜索走过的路径称为引用链
- 不可达的对象就是需要回收的
- GC Roots 包括：
  - 虚拟机栈中引用的对象
  - 方法区中类静态属性引用的对象
  - 方法区中常量引用的对象
  - 本地方法栈中 JNI 引用的对象

### 引用分类

- **强引用**：`Object obj = new Object()`，永远不会被回收
- **软引用**：内存不够才回收，`SoftReference`，适合做缓存
- **弱引用**：下次 GC 一定回收，`WeakReference`
- **虚引用**：`PhantomReference`，唯一目的就是跟踪回收，必须配合 `ReferenceQueue` 使用

## 垃圾回收算法

### 标记-清除算法

**步骤**：先标记所有需要回收的对象，标记完成后统一清除所有标记对象

**缺点**：
- 产生大量内存碎片
- 碎片过多可能导致无法分配大对象

### 复制算法

**步骤**：将可用内存分为两块，只使用其中一块；回收时将存活对象复制到另一块，然后一次性清空当前块

**优点**：
- 没有内存碎片
- 实现简单，运行高效

**缺点**：
- 内存浪费：只能用一半
- 对象存活率高时复制次数多，性能下降

**HotSpot 新生代使用**：新生代 8:1:1 划分（Eden : Survivor0 : Survivor1），大部分对象在 Eden 分配，只有 1/10 浪费。

### 标记-整理算法

**步骤**：标记存活对象，然后让所有存活对象向一端移动，直接清理掉端边界以外的内存

**优点**：没有内存碎片

**缺点**：需要移动对象，开销较大

**适用场景**：老年代，对象存活率高。

### 分代收集理论

分代收集基于两个假说：
1. **弱分代假说**：绝大多数对象都是朝生夕灭
2. **强分代假说**：熬过越多次垃圾收集的对象，越难死亡

**设计原则**：
- 新生代：对象朝生夕灭，用**复制算法**
- 老年代：对象存活率高，用**标记-清除**或**标记-整理**

## HotSpot 垃圾回收器

### Serial 收集器

- 单线程收集，收集时必须暂停所有用户线程（Stop The World）
- 客户端模式下默认新生代收集器
- 优点：简单高效，内存消耗小
- 缺点：单线程，STW 时间长

### ParNew 收集器

- Serial 的多线程版本
- 很多参数和 Serial 一样，只是多线程并行收集
- 配合 CMS 使用
- 只有在 CMS 作为老年代收集器时使用

### Parallel Scavenge / Parallel Old

- 新生代复制算法，老年代标记-整理
- **关注点：吞吐量** = 运行用户代码时间 / (运行用户代码时间 + GC 时间)
- 适合后台计算、不需要太多交互的应用
- JDK 8 默认收集器组合

### CMS (Concurrent Mark Sweep)

- **关注点：低延迟**，目标是获取最短回收停顿时间
- 老年代收集器，配合 ParNew 新生代使用
- **四个阶段**：
  1. 初始标记（CMS initial mark）- STW，只标记 GC Roots 直接关联对象，很快
  2. 并发标记（CMS concurrent mark）- 和用户线程并发执行，遍历对象图
  3. 重新标记（CMS remark）- STW，修正并发标记期间的变化
  4. 并发清除（CMS concurrent sweep）- 和用户线程并发清除垃圾

**优点**：并发收集，低停顿

**缺点**：
- 产生内存碎片
- 并发阶段会占用 CPU，降低总吞吐量
- 对 CPU 资源敏感
- JDK 9 开始标记为废弃，JDK 14 完全移除

### G1 (Garbage-First)

- 面向服务端应用，面向垃圾优先的垃圾回收器
- JDK 9 之后默认垃圾回收器
- **特点**：
  - 将堆划分为多个大小相等的 Region
  - 跟踪每个 Region 垃圾数量，优先回收垃圾最多的 Region
  - 可预测停顿时间
  - 整体是标记-整理，局部是复制，不产生内存碎片

**四个阶段**：
1. 初始标记 - STW
2. 并发标记 - 并发
3. 最终标记 - STW
4. 筛选回收 - 并发回收

### ZGC

- JDK 11 实验性，JDK 15 正式发布
- **可扩展低延迟垃圾回收器**
- 基于染色指针、读屏障、颜色标记
- 几乎所有工作都是并发执行，停顿时间不超过 10ms
- 支持 TB 级堆
- JDK 当前最新垃圾回收器，生产环境可用
